# GICv3 + R82 搭建中断系统要注意什么

本文不局限于当前仓库，而是回答两个更通用的问题：

1. 如果基于 GICv3 和 Cortex-R82 搭一个能使用中断的系统，需要注意什么
2. 如果新增一个设备的中断支持，需要注意什么

---

## 一、基于 GICv3 + R82 搭一个能使用中断的系统，需要注意什么

### 1. 先把启动链路打通

必须先确认：

- 向量表已经安装
- IRQ/FIQ 已经路由到正确的异常级
- 中断 mask 已经打开
- 栈、异常入口、现场保存/恢复是可靠的

如果这些没做好，后面设备和 GIC 配再对，也进不了中断。

### 2. 先分清三层职责

- 外设层：谁产生中断，怎么清中断
- GIC 层：谁使能、优先级、路由、触发类型
- CPU 层：怎么取 INTID、怎么回 EOI

调试时必须分层排查，不然很容易混淆问题来源。

### 3. Distributor / Redistributor / CPU interface 都要初始化

- Distributor：全局共享中断
- Redistributor：每核本地相关中断
- CPU interface：优先级门槛、Group 使能、IAR / EOI

GICv3 不是只开一个总开关就完事。

### 4. 每个核的 Redistributor 要单独考虑

- 当前核对应哪个 Redistributor
- RD 是否已经 wake up
- 多核时 SPI 路由到哪个核
- 是否允许中断迁移

这是 GICv3 和更老式中断控制器的重要区别之一。

### 5. 先定中断模型

先明确：

- 用 IRQ 还是 FIQ
- 用 Group0 还是 Group1
- 安全态 / 非安全态怎么分
- 一个系统里哪些中断属于哪组

不要写到一半才发现组别和异常入口对不上。

### 6. 优先级体系要先定规则

- 数值越小优先级越高
- CPU 的 priority mask 会过滤中断
- 抢占关系要明确
- 高优先级 ISR 里不要做重活

建议先做最简单版本：

- 统一优先级
- 先跑通
- 再逐步细化抢占策略

### 7. 触发方式必须和硬件一致

- 电平触发还是边沿触发
- 电平触发通常要先清外设状态再 EOI
- 边沿触发要避免丢事件

这个点错了，最容易出现：

- 中断风暴
- 中断只进一次
- 中断 pending 不消失

### 8. ISR 要尽量短

典型顺序：

1. 读 IAR 拿 INTID
2. 分发到设备 handler
3. 清设备中断源
4. 写 EOI
5. 尽量快返回

复杂工作应放到下半部、任务或线程里做。

### 9. 地址空间和内存属性要确认

- GIC 和设备寄存器通常是 memory-mapped I/O
- 如果启用 MMU / MPU，要设置成合适的设备内存属性
- 地址映射必须和平台手册一致

这类错误往往表面现象很乱，但根因是访问属性错了。

### 10. 要准备一套排错路径

排错时建议按链路逐层看：

1. 外设状态位有没有置位
2. GIC enable / pending / group / priority 对不对
3. CPU 有没有读到正确 INTID
4. ISR 里有没有清源
5. EOI 有没有写

## 二、如果新增一个设备的中断支持，要注意什么

### 1. 先拿到最关键的设备信息

至少要确认：

- 设备基地址
- 中断号 `INTID`
- 它是 SPI 还是 PPI
- 触发方式：level 还是 edge
- 中断状态寄存器在哪
- 清中断寄存器或清中断方法是什么

### 2. 先看设备手册，不要猜

重点确认：

- 哪个位表示中断使能
- 哪个位表示 raw status / masked status
- 是读数据自动清，还是写 1 清
- FIFO / error / timeout 是否共用中断

新增设备时，最大风险常常不是 GIC，而是误解了设备寄存器语义。

### 3. 按固定模板接入

建议固定成下面这套流程：

1. 注册设备基地址
2. 配设备内部中断源
3. 在 GIC 里设优先级 / 分组 / 路由 / 触发方式
4. 使能 `INTID`
5. 写 ISR
6. ISR 里读状态、清源、处理数据、写 EOI

### 4. 先做最小可用中断

- 先只开一个最核心中断
- 先证明能进 handler
- 再加错误中断、TX/RX 分离、超时等细节

不要一开始就把所有子中断全开。

### 5. 区分“设备使能”和“GIC使能”

- 设备没开：不会发中断
- GIC 没开：设备发了也到不了 CPU

这两个都要配。

### 6. ISR 清中断顺序要对

一般建议：

1. 先读设备状态
2. 再清设备中断源
3. 最后写 EOI

尤其对电平触发中断，顺序错了很容易重复进中断。

### 7. 并发访问要设计好

重点考虑：

- ISR 和主线程共享数据怎么保护
- 用环形缓冲区还是队列
- `volatile` 能解决什么，不能解决什么

串口、网络、DMA 这类设备最容易踩这个坑。

### 8. 多核归属要明确

- 设备中断固定给哪个核
- 是否允许迁移
- ISR 数据结构是否按核分开

多核下，建议先固定由一个核接管，跑通后再扩展。

### 9. 做好统计和日志

建议至少有：

- 进中断次数
- 最后一次 INTID
- 设备状态寄存器快照
- 错误计数

这样定位问题会快很多。

### 10. 优先复用已有模式

- 用现有简单设备中断例子做模板
- 先替换基地址、INTID、清中断逻辑
- 再逐步加入目标设备自己的特性

不要从零硬搭。

---

## 三、落地建议

如果现在真要基于当前仓库风格扩展，建议顺序：

1. 先用现有框架跑通一个最简单 SPI 设备中断
2. 先只支持单核
3. 先只用一种 Group
4. 先统一优先级
5. ISR 只做最小处理
6. 跑通后再逐步引入复杂设备逻辑

---

## 四、一句话总结

搭系统时，重点是把这条主链打通：

- 启动
- 异常入口
- GIC
- CPU interface

加新设备时，重点是搞清：

- 设备寄存器语义
- INTID
- 清中断方式
- GIC 接入模板
