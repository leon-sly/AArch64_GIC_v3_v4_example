# 当前代码覆盖了哪些中断问题

本文从两个角度总结当前仓库已经解决了什么、没有解决什么：

1. 基于 GICv3 搭一个“能处理中断”的最小系统
2. 给一个新设备接入中断支持

不展开实现细节，只记录覆盖范围。

---

## 一、这份代码怎么把“系统能处理中断”搭起来

### 1. 启动链路

- 已解决
- `src/startup.s` 完成早期启动、安装 `VBAR_EL3`、打开 SRE、清 IRQ/FIQ mask、跳转 `__main`

### 2. 异常入口

- 已解决
- `src/el3_vectors.s` 定义 EL3 向量表，FIQ 路径进入 `fiqFirstLevelHandler`，再到各示例里的 `fiqHandler()`

### 3. CPU interface 初始化

- 已解决
- `src/gicv3_cpuif.S`
- 包括：
  - `enableGroup0Ints()`
  - `enableGroup1Ints()`
  - `enableNSGroup1Ints()`
  - `setPriorityMask()`
  - `readIARGrp0()`
  - `writeEOIGrp0()`

### 4. GIC Distributor / Redistributor 初始化

- 已解决
- `src/gicv3_basic.c`
- 包括：
  - `setGICAddr()`
  - `enableGIC()`
  - `getRedistID()`
  - `wakeUpRedist()`

### 5. 多核下 Redistributor 识别

- 已解决
- `getAffinity()` 从 `MPIDR_EL1` 取当前核 affinity
- `getRedistID()` 用 `GICR_TYPER` 匹配当前核对应的 RD

### 6. 中断分组模型

- 部分解决
- 代码支持：
  - Group0
  - Group1
  - NS Group1
- 当前 `main_basic.c` 示例主要把中断都放在 `GICV3_GROUP0`
- 更复杂的系统级分组策略无

### 7. 优先级体系

- 基础解决
- 支持设置中断优先级和 CPU priority mask
- 当前示例用法较简单：
  - 中断优先级统一设 `0`
  - priority mask 设 `0xFF`
- 完整抢占策略无

### 8. 触发方式

- 基础解决
- 支持：
  - `GICV3_CONFIG_LEVEL`
  - `GICV3_CONFIG_EDGE`
- 高/低电平极性配置无

### 9. 中断处理主链

- 已解决
- 典型流程完整：
  - 读 IAR
  - 判断 INTID
  - 清中断源
  - 写 EOI

### 10. MMU / MPU / 内存属性

- 无
- 当前代码默认裸机平坦映射，按物理地址直接访问

### 11. 调度 / 下半部 / 任务系统

- 无
- 没有 OS，没有中断下半部框架

### 12. 系统化排错框架

- 基础有
- 有一些 `DEBUG` 打印和中断号打印
- 完整诊断框架无

---

## 二、这份代码怎么支持“新增一个设备的中断”

### 1. 设备基地址接入

- 已解决
- 典型方式：
  - 定义 `BASE_ADDR`
  - 再 `setXxxBaseAddress(BASE_ADDR)`

### 2. 设备寄存器抽象

- 已解决
- 例子：
  - `src/sp804_timer.c`
  - `headers/sp804_timer.h`
- 通过结构体映射寄存器布局

### 3. GIC 侧注册中断

- 已解决
- 可用函数包括：
  - `setIntPriority()`
  - `setIntGroup()`
  - `setIntRoute()`（SPI）
  - `setIntType()`
  - `enableInt()`

### 4. 区分 SPI / PPI

- 已解决
- `gicv3_basic.c` 里按 INTID 范围区分
- SPI 走 Distributor
- PPI / SGI 走 Redistributor

### 5. 扩展中断支持

- 已解决
- 支持：
  - Extended PPI
  - Extended SPI
- 有合法性检查函数

### 6. 外设自身中断配置

- 已解决，但由每个设备自己负责
- 例子：
  - SP804 在 `initTimer(..., SP804_GENERATE_IRQ)` 中打开设备侧中断

### 7. ISR 中清设备中断

- 已解决
- 例子：
  - `main_basic.c` 中 `case 34: clearTimerIrq();`
- 说明“清设备中断源”和“写 EOI”是两件事

### 8. 新设备接入模板

- 隐含解决
- 没有单独模板文档
- 但 `SP804` 例子本身就可以作为模板

### 9. 多子中断源 / 复杂状态分发

- 无
- 当前设备示例都较简单

### 10. 并发缓冲 / ISR 与线程协作

- 无
- 没有串口环形缓冲、DMA 回调、任务同步这类内容

### 11. 多核设备中断归属策略

- 基础有
- SPI 可以通过 `setIntRoute()` 路由到指定 affinity
- 更复杂的负载均衡策略无

### 12. 新设备手册 / 寄存器语义说明

- 无
- 这部分要靠目标设备自己的 TRM

---

## 三、最接近“新增设备中断支持模板”的现有代码

最适合参考的是：

- `src/main_basic.c`
- `src/sp804_timer.c`
- `src/gicv3_basic.c`
- `src/gicv3_cpuif.S`

可以抽象成这套模板：

1. 定义设备基地址和 INTID
2. 初始化 GIC
3. 配中断优先级 / 分组 / 路由 / 触发方式
4. 使能 INTID
5. 初始化设备寄存器并打开设备中断
6. 在 `fiqHandler()` 里按 INTID 分发
7. 清设备中断源
8. 写 EOI

---

## 四、一句话总结

这份代码已经解决了：

- 裸机下把 GICv3 中断主链跑通
- 接入一个简单设备中断

这份代码没有解决：

- OS 级调度
- 下半部
- 并发驱动框架
- MMU / MPU
- 复杂多核策略
- 复杂设备驱动模型
